# 프로젝트 코드 상세 설명서 (CODE_EXPLANATION.md)

이 문서는 '레시피 깃허브(Git for Recipes)' 프로젝트의 설계 철학, 아키텍처, 그리고 각 모듈의 상세 작동 원리를 설명합니다.

---

## 1. 프로젝트 철학: "요리를 코드처럼"
우리는 레시피를 단순한 텍스트가 아닌, 버전 관리와 분기가 가능한 **데이터 자산**으로 정의합니다.
- **불변성(Immutability):** 한 번 기록된 레시피 버전(Commit)은 수정되지 않으며, 오직 새로운 커밋을 통해서만 진화합니다.
- **추적성(Traceability):** 어떤 재료가 왜 바뀌었는지 히스토리를 통해 명확히 파악합니다.
- **확장성(Extensibility):** 타인의 아이디어를 '포크(Fork)'하여 나만의 창의성을 더합니다.

## 2. 아키텍처 상세 구조

### A. 데이터 및 검증 레이어 (`src/schemas`)
- **`recipeSchema.ts`**: Zod 라이브러리를 사용하여 Supabase의 JSONB 데이터를 검증합니다. 
  - **왜 Zod인가?** JSONB는 유연하지만 구조가 깨질 위험이 큽니다. Zod는 런타임에서 데이터를 강제 검증하여 '화이트 스크린' 오류를 원천 차단합니다.

### B. 비즈니스 로직 레이어 (`src/hooks`)
- **`useRecipeGit.ts`**: 레시피의 Git 핵심 로직(Fork, Commit, Version Load)을 담당합니다.
  - **단방향 데이터 흐름:** UI의 임시 상태와 DB의 확정 상태를 분리하여 데이터 오염을 방지합니다.
- **`useRecipeAI.ts`**: Gemini API를 호출하여 레시피 분석 조언을 생성합니다.
  - **프롬프트 엔지니어링:** 사용자의 수정 의도를 요리 과학적 관점에서 해석하도록 설계되었습니다.

### C. UI/UX 레이어 (`src/components` & `src/pages`)
- **`RecipeDiffViewer.tsx`**: 두 JSON 스냅샷 간의 차이점을 시각적으로 비교합니다.
- **`CommitTimeline.tsx`**: 레시피의 역사적 흐름을 타임라인으로 표시하여 탐색을 돕습니다.
- **`RecipeEditorPage.tsx`**: 모든 기능이 집약된 메인 대시보드입니다.

### D. 인프라 레이어
- **Supabase**: PostgreSQL의 강력한 JSONB 처리 능력과 RLS(보안 정책)를 활용합니다.
- **Cloudflare R2**: 조리 단계별 고해상도 이미지를 체계적인 경로 규칙(`recipes/{id}/{commit}/...`)에 따라 보관합니다.

## 3. 개발 가이드 및 준수 사항
- **엄격한 타입 시스템:** `any` 사용을 금지하고 모든 데이터는 인터페이스화합니다.
- **상세한 주석:** '무엇'을 하는지보다 **'왜'** 이렇게 설계했는지를 주석으로 남깁니다.
- **자동 검증:** 커밋 전 `npm run validate`를 통해 코드 품질을 유지합니다.
