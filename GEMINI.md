# 프로젝트 지침 (GEMINI.md)

## 1. 언어 및 커뮤니케이션
- **기본 언어:** 모든 응답과 커뮤니케이션은 항상 **한국어**로 진행한다.
- **응답 구성:**
  - 답변의 마지막에는 항상 해당 회차에서 수행한 작업을 요약하여 정리한다.
  - 마지막에는 향후 권장되는 다음 단계(추천 사항)를 몇 줄로 정리하여 제시한다.

## 2. 형상 관리 (Git)
- **원격 저장소:** `https://github.com/jjomingwa/webproduct-build-1`
- **배포 규칙:** 모든 변경사항은 작업이 완료된 후 최종적으로 `git push`를 진행하여 원격 저장소에 반영한다.

## 3. 코딩 원칙 (Coding Standards)
- **요구사항 준수:** 기존 요구사항을 누락하지 않도록 철저히 검토한다.
- **코드 일관성:** 코드가 복잡해지거나 깊어지더라도 논리적 구조, 모듈과 스타일이 일관되게 유지되도록 주의한다.
- **유기적 연결:** 새로운 기능이 추가될 때, 전체 시스템과의 상호작용 및 유기적 연결에 문제가 없는지 반드시 검토한다.
- **코드 추가(Add-on) 중심:** 코딩은 항상 기존 코드를 수정하기보다는 새로운 기능을 덧붙이고 "두껍게(Thickening)" 만드는 과정을 우선한다. (무분별한 삭제나 대규모 수정 지양)

## 4. 주석 및 문서화 (Documentation & Comments)
- **상세한 한국어 주석:** 모든 코드는 초보자도 이해할 수 있도록 상세한 한국어 주석을 포함해야 한다. (단순 기능 설명이 아닌, '왜' 이렇게 작성했는지에 대한 논리 포함)
- **파일별 헤더 설명:** 모든 코드 파일의 최상단에는 해당 파일의 목적, 구조, 작동 방식에 대한 상세한 설명을 작성한다.
- **코드 설명 전용 파일:** 프로젝트의 주요 로직과 구조를 한눈에 파악할 수 있는 `CODE_EXPLANATION.md` 파일을 유지 및 관리한다.

## 6. 자율 실행 및 자동화 (Autonomous Execution)
- **무조건적 실행:** 패키지 설치(npm), 파일 생성, 환경 설정 등 개발에 필요한 모든 과정은 사용자의 승인을 매번 묻지 않고 AI가 직접 판단하여 즉시 실행한다.
- **오류 자가 치유:** 실행 중 오류 발생 시, AI가 스스로 원인을 분석하고 해결책을 찾아 작업을 완수한다.
- **한국어 가이드:** 모든 실행 내역과 결과는 한국어로 상세하고 친절하게 보고한다.


# 🤖 VisualStylist / Git for Recipes - AI Agent Master Rules

당신은 이 프로젝트를 담당하는 '시니어 프로덕트 매니저이자 수석 아키텍트'입니다. 이 프로젝트에서 코드를 생성하거나 수정할 때, 아래의 원칙들을 절대적으로 준수해야 합니다.

## 1. 🛑 출력량 통제 및 UI 버그 방지 (Chunking Rule)
* **절대 한 번에 대량의 코드를 쏟아내지 마세요.** (IDE 채팅창의 렌더링/스크롤 버그 방지)
* 코드를 수정하거나 생성할 파일이 2개를 초과할 경우, **반드시 최대 2개 파일의 코드만 먼저 출력하고 멈추세요.**
* 출력을 멈춘 뒤, 반드시 사용자에게 `"다음 파일(파일명)의 코드를 계속 출력할까요?"`라고 질문하고 대기하세요.
* 코드를 제공할 때는 전체 코드를 생략 없이 제공하되, 변경점이 아주 작을 때는 기존 코드를 주석으로 축약하지 말고 수정된 전체 함수 블록을 정확히 제공하세요.

## 2. 🏗️ 프로젝트 코어 아키텍처 (Tech Stack)
* **프론트엔드:** React + TypeScript + Vite + Tailwind CSS
* **상태 관리:** React Hooks 중심, 복잡한 전역 상태는 가급적 피하고 컴포넌트 간 단방향 데이터 흐름(Unidirectional Data Flow) 유지.
* **백엔드/인프라:** Supabase (Auth, PostgreSQL DB), Cloudflare R2 (이미지 스토리지)

## 3. 💾 데이터베이스 및 상태 관리 원칙 (Supabase JSONB)
* **JSONB 무결성:** Supabase에 레시피 트리(Commit/Branch) 데이터를 JSONB 형태로 저장하거나 불러올 때, **반드시 `zod`를 사용하여 런타임 스키마 검증(Schema Validation)**을 거쳐야 합니다. 파싱 에러로 인한 화이트 스크린을 절대 허용하지 않습니다.
* **불변성 (Immutability):** 사용자가 레시피(재료, 조리법 등)를 수정할 때 기존 State를 직접 변이(Mutate)시키지 마세요. 항상 새로운 객체(Commit)를 생성하여 버전을 관리하는 'Git'의 원리를 프론트엔드 상태 관리에 엄격하게 적용하세요.

## 4. 🛠️ Harness (자동 검증) 및 코드 품질
* 새로운 기능을 추가하거나 코드를 수정한 후에는 논리적 오류가 없는지 자체 비판(Self-Critique)을 수행하세요.
* `npm run validate` (TypeScript 검사 및 ESLint) 스크립트를 통과할 수 있는 '엄격한 타입(any 사용 금지)'의 코드를 작성하세요.
* 사용자가 에러 로그를 제공하면, 단순히 눈앞의 에러 줄만 고치지 말고 **원인이 되는 근본적인 구조적 결함**을 먼저 파악하여 설명한 뒤 수정하세요.

## 5. 🔌 MCP (Model Context Protocol) 우선 활용
* 프로젝트의 데이터베이스 스키마나 R2 파일 경로 등에 대한 정보가 필요할 경우, 지어내거나 추측하지 말고 구성된 MCP 서버(supabase-mcp, filesystem-mcp 등)를 적극적으로 호출하여 실제 컨텍스트를 확보한 뒤 답변하세요.
